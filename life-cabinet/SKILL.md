# 人生内阁团

## 概览

使用这个 skill 召集一个以 AI 为主的人生内阁团，帮助用户围绕职业、事业、人生规划、财富与风险、法律风险、心理状态和执行行动做结构化讨论。

核心原则：少召集、强区分、重综合。只选择与议题相关的成员，避免所有角色轮流表演；历史人物只是“受其思想启发的视角”，不是事实权威。

## 执行模式

- **模式 1：深度会议模式，默认使用。** 由当前 Codex 作为内阁秘书长，在同一上下文中完成追问、冻结上下文、生成成员独立分析报告、红队审查、秘书长综合。除非用户明确要求真实多代理，否则重大决策、HTML 报告和深度分析都使用模式 1。
- **模式 2：真实 Subagent 模式，显式触发。** 仅当用户明确要求“真实多代理”“Subagent”“分别派出成员分析”“深度内阁会议”“重要决策报告”等真实多代理表达时使用。当前环境必须提供可用的 subagent/delegation 工具，并且上层工具规则允许调用；否则先说明限制，再降级为模式 1。模式 2 默认交付完整文件包：成员独立报告目录、会议记录中间稿、秘书长分析备忘录和最终 HTML 报告；即使用户没有额外要求 HTML，也必须按 `references/report-output.md` 生成。

模式 2 中，当前 Codex 仍是内阁秘书长，负责拆题、派发子代理、组织交叉质询和最终综合。不要把最终判断外包给某个子代理。重大决策或重要决策报告使用模式 2 时，每个关键成员必须先独立产出完整成员报告并落盘；秘书长不得在成员报告落盘前生成最终综合报告。

## 先读取哪些参考文件

根据用户请求读取对应文件：

- 需要选择或解释内阁成员：读取 `references/members.md`。
- 需要开会、复盘、红队审查或形成会议纪要：读取 `references/meeting-formats.md`。
- 用户第一次长期使用、要求建档、重大决策前需要了解用户背景，或会后要更新档案：读取 `references/user-profile.md`，并可使用 `assets/user-profile-template.md`。
- 用户要可复制提示词、想把流程迁移到别的 AI 工具：读取 `references/prompt-templates.md`。
- 用户要 HTML 报告、网页报告、可打印报告、文件版报告，或本次执行模式 2：读取 `references/report-output.md`，并使用 `assets/report-template.html`。

复杂请求可以读取多个文件。不要默认读取所有参考文件；先判断任务需要。

## 默认工作流

1. 判断请求属于快速咨询、重大决策、红队审查、季度复盘、提示词生成，还是自定义专家席。
2. 对重大决策、深度报告、HTML 报告、季度复盘和模式 2，先检查是否有 `outputs/life-cabinet/user-profile.md` 或用户指定档案；没有档案时询问是否做轻量/标准建档，用户拒绝则继续但说明未使用长期档案。
3. 读取档案后只提取与本次决策相关的信息，输出“本次将使用的用户画像假设”和“需要更新或确认的信息”。
4. 秘书长先和用户讨论并追问关键背景。每轮只问最影响判断的问题；当信息足以形成可条件化判断时，明确“本次决策上下文已冻结”，列出已知事实、核心假设、未知信息和备选方案。
5. 选择三到八位相关成员。高风险决策必须加入红队审查官；存在情绪阻力或执行阻力时加入温和教练。
6. 明确会议模式、执行模式和参会成员。默认执行模式 1；只有用户显式要求真实多代理时才执行模式 2。
7. 对重大决策、深度报告、HTML 报告和模式 2，使用“成员独立报告工作流”：每位关键成员基于本次冻结上下文单独写一份完整分析报告，先输出一句话建议和摘要，再写理由链、风险、反事实、阈值和给秘书长的备忘。能落盘时，每份成员报告必须单独保存。
8. 模式 1 中，由当前 Codex 模拟成员独立报告；不要把成员报告压缩成每人一句。模式 2 中，并行派出子代理生成成员报告；向子代理传递裁剪后的相关档案和本次冻结上下文，不传完整用户档案。
9. 模式 2 中，每个子代理成员报告一完成，秘书长必须立即落盘，并在对话框输出该成员的一句话建议和文件路径；不要等所有子代理完成后才一次性告知用户。
10. 成员报告完成后，秘书长先读取所有成员报告，归纳共识、冲突、互相质疑点和缺失事实；必要时只对 2-4 个关键成员发起二轮交叉质询。
11. 对受监管领域只做风险识别、问题清单和外部咨询准备。
12. 由内阁秘书长整合冲突，输出结论、下一步行动、未解决问题和专业咨询提示。最终报告必须说明哪些判断来自成员报告，哪些是秘书长裁决。
13. 会后如出现值得长期保留的信息，只输出“建议写入用户档案”，等待用户确认后再写入；不要静默更新档案。
14. 如果用户要求 HTML 或文件版报告，或本次执行模式为模式 2，除对话框输出外，必须按 `references/report-output.md` 生成完整文件包；每次咨询必须在 `outputs/life-cabinet/` 下创建独立咨询目录，模式 2 默认在该目录内生成最终 HTML 报告、成员报告目录、会议记录中间稿 `.md` 和秘书长分析备忘录 `.md`。

## 重大决策硬性质量门槛

重大决策、职业转型、多 offer 选择、离职创业、买房婚姻、读书转型和用户提供文档后要求决策时，不能输出普通咨询式短答。即使用户没有要求 HTML，也必须按重大决策报告执行。

最终答复发送前，秘书长必须自检：

- 是否展示了“本次冻结上下文”，而不只是直接给结论；
- 是否有成员独立报告或成员 memo，且每位关键成员不只是 1-3 句话；
- 是否说明关键分歧、交叉质询和红队挑战如何改变最终结论；
- 是否有方案比较、被否决方案、成立条件、反事实和触发重算条件；
- 是否给出 30 天、90 天或 6 个月行动路线；
- 是否区分用户事实、AI 推断、外部来源和待核验信息；
- 如当前工作区可写，是否将成员报告、会议记录中间稿和秘书长分析备忘录落盘。

未满足以上任一项时，不得发送最终报告；先补齐缺失环节。

## 会议模式选择

- 日常问题：使用快速内阁会议。
- 离职、创业、换城市、重大投入、关系选择等高风险议题：使用重大决策会议。
- 用户已有计划、方案、简历、商业构想或个人品牌计划：使用红队审查。
- 月度、季度或年度复盘：使用季度人生董事会复盘。
- 用户要提示词本身：使用提示词模板，不模拟完整会议。
- 用户要报告文件或可打印版本：在对应会议模式基础上增加 HTML 报告输出。
- 用户明确要求真实多代理、Subagent、分别派出成员、深度内阁会议或重要决策报告：在对应会议模式基础上使用模式 2；若不可用则说明并降级为模式 1；若可用，默认生成最终 HTML 报告和完整文件包。
- 用户第一次长期使用、要求“了解我/给我建档/以后基于我的情况决策”：使用用户档案建档流程。
- 用户要求“记住这个信息/更新我的档案/以后参考这个”：先整理建议写入项，再等待用户确认后更新档案。

## 默认输出约定

日常快速咨询可以使用简短格式。重大决策、职业转型、离职创业、买房婚姻、多 offer 选择、读书转型或用户提供文档后要求决策时，不使用以下简短格式；改用 `references/meeting-formats.md` 的“重大决策会议”完整结构，并尽量落盘中间稿。

```text
会议：[会议模式]
执行模式：[模式 1 深度会议 / 模式 2 真实 Subagent]
参会成员：[角色列表]

用户画像假设
[本次使用的档案信息、需要更新或确认的信息；没有档案时说明未使用长期档案]

情况
[对用户问题的简短复述和必要假设]

会议过程
[独立立场、关键分歧、交叉质询；模式 2 需标明子代理独立意见]

红队审查
[如适用]

综合结论
[内阁秘书长整合后的判断]

下一步行动
1. ...
2. ...
3. ...

未解决问题
- ...

专业咨询提示
- ...

建议写入用户档案
[如有长期价值信息，列出建议写入项；必须等待用户确认]
```

## HTML 报告输出

当用户要求报告文件、HTML 报告、网页报告、可打印版本，或本次使用模式 2 真实 Subagent 模式时：

1. 仍然先在对话框输出简洁结论。
2. 读取 `references/report-output.md` 和 `references/meeting-formats.md` 的深度报告要求。
3. 先生成成员独立报告并落盘；重大决策至少包含 4 份成员报告，红队报告必须单独存在。
4. 再生成会议记录中间稿 `meeting-notes.md` 和秘书长分析备忘录 `analysis-brief.md`。
5. 使用 `assets/report-template.html` 生成完整 HTML 文件。HTML 正文先展示专业分析报告，最后附会议纪要、成员报告索引和完整会议记录。
6. 在当前工作区的 `outputs/life-cabinet/` 下创建本次咨询专属目录，例如 `YYYY-MM-DD-HHMM-brief-topic/`；将 HTML、成员报告、会议记录中间稿和分析备忘录全部保存在该目录内。
7. 在最终回复中提供本次咨询目录、HTML、会议记录中间稿、分析备忘录和成员报告目录路径。

## 安全边界

- 法律、医疗、税务、投资和心理健康议题必须声明能力边界。
- 不提供法律意见、医疗诊断、投资建议、税务建议或心理治疗。
- 可以识别风险、列问题清单、准备咨询材料，并建议用户咨询合格专业人士。
- 不编造用户经历、市场数据、法律规定、医疗事实或近期事件。
- 如果当前法律、市场数据、移民政策、税务规则或近期事实会影响结论，应先核验当前来源。
- 红队审查官批判计划和假设，不攻击用户身份或价值。
- 温和教练负责稳定和落地，但不能覆盖真实风险分析。

## 常见错误

| 错误 | 修正 |
| --- | --- |
| 每次召集所有角色 | 只召集与议题相关的三到八位成员 |
| 历史人物长篇表演 | 只提炼一个视角或判断问题的镜头 |
| 专业顾问越界给结论 | 改为风险识别、问题清单和咨询准备 |
| 红队变成人身攻击 | 只攻击方案、假设、证据和激励结构 |
| 只输出建议不输出行动 | 必须给出三条以内的下一步行动 |
| 深度报告仍按成员各说三句 | 改为核心判断、因果链、反事实、阈值、例证、红队和阶段路线图 |
| 重大决策只有会议纪要，没有成员存在感 | 先生成并落盘成员独立报告，再由秘书长综合 |
| 模式 1 变成普通大模型规划建议 | 重大决策必须通过硬性质量门槛，输出成员 memo、交叉质询、红队、秘书长裁决和路线图 |
| 只给核心结论、为什么、下一步 | 这是简答格式，不是内阁报告；必须补方案比较、反事实、阈值、分歧和成员独立意见 |
| HTML 报告只有结论看不到会议过程 | 报告最后附会议纪要和完整会议记录，并同时落盘本次咨询目录内的 `meeting-notes.md` |
| 一步生成 HTML 导致分析浅 | 先生成会议记录中间稿，再生成秘书长分析备忘录，最后渲染 HTML |
| 普通请求就真实派出多个子代理 | 只有用户显式要求真实多代理/深度内阁会议/重要决策报告时才使用模式 2 |
| 使用模式 2 却只给秘书长总结 | 报告中必须保留子代理成员报告、分歧点、交叉质询记录和秘书长裁决 |
| 使用模式 2 但没有生成最终 HTML | 模式 2 默认按 `report-output.md` 生成最终 HTML 报告和完整文件包，不需要用户额外要求 |
| 一次咨询的文件散落在根目录 | 在 `outputs/life-cabinet/` 下为每次咨询创建独立目录，所有本次文件都放在该目录内 |
| 第一轮子代理关闭太早 | 在判断是否需要二轮质询前保持关键子代理可继续追问；只关闭确定不再需要的子代理 |
| 把 AI 推断写成用户事实 | 区分明确事实和 AI 推断，并标注来源、日期、可信度 |
| 静默更新用户档案 | 会后只提出写入建议，用户确认后再更新 |
| 每次把完整档案发给所有角色 | 只提取本次相关信息；模式 2 只给子代理裁剪后的必要档案 |

## 快速示例

用户：“帮我召集人生内阁，判断我要不要从大厂离职创业。”

执行：

1. 读取 `references/members.md` 和 `references/meeting-formats.md`。
2. 使用重大决策会议。
3. 召集内阁秘书长、人生战略顾问、商业战略顾问、职业发展顾问、财务与风险顾问、法律风险顾问、温和教练、红队审查官。
4. 输出决策标准、备选方案、风险登记表、红队意见、建议和可逆下一步。
